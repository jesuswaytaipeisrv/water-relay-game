## 2026-06-22（二）：提水人物改打水驅動 + 沙漠賽場

### 回饋與調整

依使用者回饋重做提水灌溉的人物與賽場氛圍：

- **提水人物**：改為雙手各提一桶（🪣🚶🪣）的卡通角色，整組左右搖晃、人物上下顛簸（搖搖晃晃前進）。
- **打水驅動移動（核心改動）**：跑者位置改由 `waterUnits` 推導，不再用時間軸。一趟行程＝一桶水（`bucketCapacity` 次打水），前半段提水往終點、中段倒水、後半段空手返回。因此**有人打水才前進、沒人按就停在原地，回程同理**。
- **面向修正**：去程面右（朝終點）、回程面左（朝起點）；`🚶` 預設面左，去程以 `scaleX(-1)` 鏡像。
- **倒水誇張化**：人物前傾甩桶（`pour-tip`）並潑出水花朝小人飛去（`splash-fly`），終點落雨改為較多較大的水滴。
- **小人成長**：依澆水比例由小慢慢變大（`transform` 過渡），每次倒水時臉部蓄力後向上彈跳放大（`water-pop`），灌溉「有感」。
- **沙漠賽場**：賽場改乾枯沙漠主題（暖沙底色、乾裂紋理、邊角熱浪），賽道散落石頭 🪨 與仙人掌 🌵；已澆水的土地由左往右回春（轉綠）代表灌溉進度。

### 技術重點

- 移除舊的時間軸接力佇列：`deliveryEvents`、`RELAY_TIMING`、`queueDelivery`、`queuePendingDeliveries`、`isPouringAtFinish`、`relayMarkup` 等全部移除；狀態版本升為 6（升級時自動丟棄舊欄位，保留 `waterUnits`）。
- 勝利條件改以澆水次數：`pours = floor(waterUnits/bucketCapacity + 0.5)`，達 `5 × growthStages` 即勝；達標門檻約 `(maxGrowth − 0.5) × bucketCapacity` 次打水（與舊版接近）。
- 賽場改「建一次、之後只更新樣式」：跑者與小人是常駐 DOM，位置只透過 `style.left` + CSS 過渡平滑滑動，不再因每 250ms 重繪而中斷搖晃／成長動畫。

### 驗證

- `node --check app.js` 通過；CSS 無殘留舊類別。
- 以 Chrome headless（CDP 裝置模擬）截圖驗證：桌面與 390px 手機賽場無水平溢位；去程跑者面右、回程面左、倒水跑者在終點潑水並落雨、小人依進度大小不同；沙地石頭與仙人掌、澆水回春皆正確。
- 程式化驗證勝負流程：`running →`（偵測達標）`finishing`（停在終點潑水約 1.7s）`→ finished` 並顯示正確獲勝隊伍遮罩。
- 快取版本更新為 `styles.css?v=20260622-2` 與 `app.js?v=20260622-2`。

## 2026-06-22：視覺與手機體驗改版（接手 Codex 版本）

### 背景

接手 Codex 完成的 `bucket` 倉庫。遊戲機制（三隊自動分隊、雙桶接力、終點小人成長、Firebase 即時同步、QR Code、結算與下一輪）完整且運作正常，但回饋指出兩個問題：**美術不好看**、**手機操作體驗差**。遊戲邏輯與 Firebase 同步全部保留，只重做視覺層。

### 核心問題

- 角色（接力跑者、起點群眾、終點小人）全部以多層 CSS `<span>` 定位拼貼（頭、頭髮、蝴蝶結、手臂、水桶皆為方塊），在真實裝置上比例易歪、不精緻，是「不好看」的主因。
- 主持頁賽場在手機上 `min-width: 680px` 強迫橫向捲動，三隊賽道擠成一團。
- `.player-view { display: grid }` 會覆蓋 `hidden` 屬性，導致主持頁也露出玩家面板（真實 bug）。

### 本次修改

- **角色改用 emoji**：終點小人 🧒 依澆水比例平滑放大、長大後變 🧑✨；接力跑者 🏃🪣、倒水 🧍💦；起點群眾與隊員名單以 emoji 頭像呈現。無圖片資產、跨裝置一致。
- **重寫 `styles.css`**：明亮的「澆水成長」主題、卡片化版面、隊色主題化的玩家打水按鈕與成長進度條。
- **賽場改為每隊一條橫向賽道（左取水 → 右灌溉）**，手機上自動垂直堆疊、不再橫向捲動。賽道底層加成長進度填色，倒水時終點落雨、小人彈跳。
- 修正 `[hidden] { display: none !important; }`，解決玩家面板在主持頁外露問題。
- 接力方向由「右→左」改為「左→右」，跑者依 `--pos` 定位、回程以 `scaleX(-1)` 翻面。
- 新增 `prefers-reduced-motion` 降低動畫。
- 快取版本更新為 `styles.css?v=20260622-1` 與 `app.js?v=20260622-1`。

### 驗證

- `node --check app.js` 通過。
- 以 Chrome headless（CDP 裝置模擬，真實 390px 行動寬度 + 桌面 1280/1366）截圖驗證：主持頁桌面/手機無水平溢位、玩家加入頁、打水頁（隊色主題 + 進度條）、進行中賽道（跑者攜桶、倒水落雨、小人成長）、結算畫面（🏆 + 下一輪）皆正確呈現。
- 修正過程發現 headless `--window-size` 最小渲染寬度約 500px 會造成截圖誤判溢位，改用 CDP `Emulation.setDeviceMetricsOverride` 取得真實行動寬度。

### 待辦

- 正式活動前以兩到三支實機手機完整演練加入、分隊、倒數、連點、接力、結算與下一輪。

## 2026-06-19：專案建立

### 目標

建立新的手機分隊打水遊戲。玩家快速點擊打水，水量由右向左流入小水桶；小水桶裝滿後自動倒進大水桶，最快累積到目標桶數的隊伍獲勝。

### 技術選型

- 純 HTML、CSS、ES Module JavaScript，沒有建置步驟，適合 GitHub Pages 或 Zeabur 靜態網站。
- Firebase Realtime Database 為選配的跨手機同步層；未設定時使用瀏覽器內示範模式。
- 以隊伍累積 `waterUnits` 作為唯一計分來源。小水桶填充量為餘數，大水桶進度為整除結果，可避免多人同時倒水時發生雙重計分。

### 已完成

- 主持台可設定每小桶需水量、勝利所需大桶數與倒數，並開始或重設比賽。
- 玩家可輸入名字、選擇隊伍，使用大面積按鈕打水。
- 兩隊均有河流、打水、移動水桶、小水桶與大水桶的視覺狀態。
- 比賽結束後顯示勝隊與完整排名；主控台可重新開始下一輪。
- Firebase 未設定時，明確顯示示範模式，避免誤以為不同手機已同步。

### 已完成驗證

- `node --check app.js`、`node --check firebase-config.js` 與 `node --check firebase-config.example.js` 通過。
- `firebase-database.rules.json` 已用 Node JSON parser 驗證。
- 本機 `python3 -m http.server 5175 --bind 127.0.0.1` 回傳 `HTTP 200`，並確認 HTML 包含主持、玩家點擊與 ES Module 載入點。
- 已檢查無 Firebase 時的同步分支、累積水量換算、大桶勝利判定、重設與玩家離隊邏輯。
